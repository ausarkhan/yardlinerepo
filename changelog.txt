YardLine Web — Phase 4: Stripe Payments (status — partially live, env-blocked)

- Database migrations applied to the live Supabase project:
  - 0001_yardtix_orders_tickets.sql — yardtix_orders / yardtix_tickets ledger: APPLIED.
  - 0003_phase4_payments.sql — checkout_session_id + currency, bookings.payment_status
    CHECK, stripe_webhook_events idempotency ledger: APPLIED.
  - 0002_booking_lifecycle.sql — NOT applied as a migration (it errored with 42710 because
    0003 ran first and 0002's `%status%` constraint matcher is ambiguous once
    bookings_payment_status_check exists). Resolved by direct verification instead — see below.
- Booking constraints confirmed correct on the live DB:
  - bookings_status_check allows: requested, pending, pending_provider_review, confirmed,
    declined, cancelled, checkout_created, completed, refund_pending, refunded.
  - bookings_payment_status_check allows: none, authorized, captured, failed, refunded.
- Phase 4 API routes are mounted and auth-gated (all return 401 without a valid Supabase token):
  POST /api/checkout/event-session, POST /api/checkout/booking-intent,
  GET /api/checkout/session/:id/status, POST /api/bookings/:id/accept,
  POST /api/bookings/:id/decline, POST /api/payments/connect/onboard.
- Stripe webhook endpoint exists: POST /api/webhooks/stripe (signature-verified, idempotent
  via stripe_webhook_events).
- CURRENT BLOCKER: the running backend is missing or not loading STRIPE_SECRET_KEY and/or
  STRIPE_WEBHOOK_SECRET, so POST /api/webhooks/stripe returns 503 "Webhooks not configured"
  instead of 400. The full payment happy-path (checkout -> webhook -> mint tickets; authorize
  -> capture/decline) cannot run end-to-end until the backend env is fixed and restarted.
  Fix + verification steps documented in docs/phase-4-payments.md.
- Full Stripe live E2E intentionally deferred; next platform features proceed without it.

YardLine Web — Phase 5: Ticket Check-in (QR scan + manual entry)

- Closes the events loop using the existing yardtix_tickets.qr_token / used_at columns — no
  new migration required. Runs through the service-role backend because RLS only lets a
  holder update their own ticket, not a host.
- Shared contract added to backend/src/types.ts: CheckinScanRequest, CheckinScanResponse,
  CheckinResultCode (ok / already_used / not_found / wrong_event / not_authorized /
  invalid_status), CheckinTicket, CheckinSummaryResponse.
- Backend (backend/src/routes/checkin.ts, mounted at /api/checkin):
  - POST /api/checkin/scan — host-gated; finds a ticket by qr_token or ticket_number,
    verifies the caller hosts the ticket's event, admits it (status='used', used_at=now) or
    returns the precise domain result. holder_name joined best-effort from profiles.
  - GET /api/checkin/event/:id/summary — host-only roster + counts (total / checked_in /
    remaining) over admissible (confirmed|used) tickets.
  - Auth-gated (401 verified live), typecheck clean.
- Webapp:
  - Holder tickets now render a REAL scannable QR (qrcode.react) of qr_token, with the
    ticket_number beneath for manual entry; RSVPs keep the door-stub placeholder.
  - Host-only /event/:id/checkin page: camera QR scanner (@zxing/browser, degrades to manual
    entry on permission-denied), manual ticket-number entry, per-scan result card, live
    roster/counts with progress, and same-token debounce.
  - Host entry points added on the event detail host bar and dashboard event rows.
  - useCheckin hook (summary query + scan mutation, Supabase bearer token); typecheck + build clean.
- Note: live camera decode unverified in this environment; manual-entry path, host guard, and
  all types/build verified.
